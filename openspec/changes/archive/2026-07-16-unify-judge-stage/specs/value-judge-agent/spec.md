## MODIFIED Requirements

### 需求:判分输入须以正文与来源 grounding

系统在调用 Value Judge 判分时，**必须**把代表 raw_item 的**正文（`content`）与来源（`source`）连同标题一并作为判分输入**（当正文经 source-content-enrichment 补全后可用时）；**禁止**退化为「仅标题」判分——除非正文补全失败致正文仍为空，此时才回退仅标题并如实以标题为唯一依据。

理由：仅凭标题判分会使 `is_ai_related` 与各项评分脱离真实内容（如仅凭标题 `Qualcomm Linux 2.0` 难以稳定判非 AI）。以补全后的正文 grounding 后，`is_ai_related` 与重要性/开发者相关性判定才有事实依据。判分输出仍必须经既有 Zod schema 校验（含 mojibake 守卫），失败重试/降级口径不变。

**「须含正文」是对所有被判事件的普遍约束，故补全必须内联在判分入口本身（本需求修改点）**：本要求作用于 `scoreUnscoredEvents` 将判的**每一个**事件（`importance_score IS NULL AND merged_into IS NULL`）。该普遍约束**禁止**依赖「补全工作集与判分工作集口径对齐」这类**跨模块约定**来兑现——判分入口是**共享的**（日报链 `runDailyWorkflow()` 与实时告警链 `alert-scan` 今天已各调一次，未来还会有第三个调用方），而评分是**一生一次**：任何一条**只判分不补全**的链都会把事件**永久踢出补全工作集**，使被它先判的事件退化仅标题、且**再也无法回灌**（评分不重跑）。

故正文补全**必须**内联在 `scoreUnscoredEvents` 之内、位于该事件**原子 claim 成功之后、送 LLM 之前**（见 source-content-enrichment「补全必须是判分入口内的第一步」）。由此「凡被判事件皆先补全（除非补全失败）」由**单一函数内的语句顺序**结构化保证，而非靠调用方遵守约定；`scoreUnscoredEvents` 的**每一个**调用方自动满足它，**不可能漏**。

**载入正文/来源的具体改动点（勿误当仅签名改动）**：`judgeRawItem` 已接受可选 `content`/`source` 入参，故实工作在**调用侧**——`scoreUnscoredEvents` 的候选 SELECT **必须** left join `raw_items`（经 `representative_raw_item_id`）载入 `content` 与 `source`，并**一并取出** `raw_items.id` / `canonical_url` / `url` 与空判定**投影列** `isEmpty` 供内联补全使用（**同一个 SELECT**，不另建第二份工作集查询）；claim 成功后先补全、再以补全后的 `content` 调 `judgeRawItem`。只改函数签名而不改 SELECT，或把补全留在调用链里编排，都会使 grounding 静默空转。

**补全的正文必须经【返回值】进入本次判分输入，只写库等于没做（绝不可省）**：候选 SELECT 是**先把 `content` 读进内存**的，补全的 `UPDATE raw_items` **不会**改写内存里那个值。故单条补全函数**必须**返回补全后的正文（`string | null`），调用侧**必须**把该返回值（而非 SELECT 时读到的旧空值）作为 `judgeRawItem` 的 `content` 入参。若补全函数只返回成功/失败标志、调用侧沿用旧值，则 **DB 被补全了、而这一次判分仍是 title-only**——评分**一生一次**，这条事件**再无第二次机会**，本变更的全部目的落空，且 DB 里那行补全好的正文**看起来一切正常**。相应地，本条的验证**必须**断言 **`judgeRawItem` 收到的 `content` 入参**等于补全后的正文；**只断言「`raw_items.content` 被写入」的测试对本条毫无证伪力**（它在缺陷存在时同样为绿）。

**补全的注入 seam 必须随补全一起迁进判分入口（绝不可省）**：补全的**能力**下沉到 `scoreUnscoredEvents` 之后，补全的**可注入性**必须同步下沉——`scoreUnscoredEvents` 的选项对象**必须**含补全选项（`enrich`：`fetchImpl` / `resolve` / `maxRedirects` / 日志 sink），并在调单条补全函数时透传；**禁止**删除补全在调用链上的旧消费者而不为其注入口安排新落点。否则调用链上既有的无网络测试桩会变成「仍在传、无人读」的死参数，补全在判分入口内部悄然走真实 `fetch` + 真实 DNS，而**测试仍全绿**（补全失败被 fail-open 吞成一次仅标题回退）。相应地，调用 `scoreUnscoredEvents` 的**每一个**测试**必须**经该选项注入无网络桩——本条的验证不能只依赖「测试通过」（缺陷存在时它同样通过）。

**「测试不触网」必须由结构性守卫兜底，禁止仅靠逐文件注入桩（绝不可省）**：补全的**默认**外部 I/O 实现——**默认 `fetchImpl` 与默认 `resolve`（`dns.lookup`）二者皆是**——**必须**在测试环境下（`process.env.VITEST`）**直接抛错**。守卫**禁止只挂在 `fetchImpl` 上**：SSRF 守卫的主机校验发生在 `fetchImpl` **之前**、且它调用真实 `dns.lookup`，只挡 HTTP 时，任何未注入 `resolve` 的测试**照样对候选的真实域名发出真实 DNS 查询**。

**抛错位置必须钉死在「默认实现的函数体内」**（与仓内既有守卫同形：`src/kb/embed-clean.ts` 的 `defaultEmbedMany`、`src/mr/curation/telegram-callback.ts` 的 `startApprovalBot`），**禁止**抛在选项解析处——后者会把「本来就不走补全路径、既不该触网也不该失败」的测试一并打红。**代价必须一并承认**：函数体内抛出的错会被补全的逐条 try/catch 吞成一次补全失败（失败计数 +1、事件照常仅标题判分），故**「测试全绿」绝不构成「零出站」的证据**——那正是本能力的失效模式（静默降级、报告成功）。「测试零出站」**必须**另由一条**可机械核验**的手段证明：断言**未注入桩时默认 `fetchImpl` / 默认 `resolve` 的调用次数为 0**（spy），或断网跑一遍全量测试并核补全失败计数为 0。

逐个测试文件注入无网络桩是**纪律**——它靠「调用方枚举」兑现，而枚举必然漏（真实 URL 恒可解析：`example.com` 是 IANA 的可解析域名，过 SSRF 守卫、真发请求，其页面无 `og:description` ⇒ 补全按失败计 ⇒ 被逐条 try/catch 吞掉 ⇒ **漏掉的那个文件照样全绿**）；默认实现上的守卫是**结构**——新增的第 N 个调用方**漏不掉**。注入桩仍**必须**逐个补齐（测试需主动控制补全的命中/失败/样板分支），但**兜底不得是纪律**。

**补全失败绝不阻止判分（fail-open，绝不可省）**：补全是 best-effort（og 缺失 / 抓取失败 / 命中全站样板 / SSRF 拒绝均按失败计）。补全失败时该事件**必须**照常送 LLM 判分、以标题为唯一依据；**禁止** fail-closed 地跳过、延后或阻止其判分——否则一次抓取失败会让该事件**永不评分、永不推送**，比仅标题判分更糟。

#### 场景:有正文时判分吃到正文与来源
- **当** 某事件经判分入口内联补全后 `content` 非空（或其代表 raw_item 本就自带正文），流水线调 Value Judge 判分
- **那么** 判分输入包含标题 + 正文 + 来源三者（补全所得正文经补全函数的**返回值**传入，不是候选 SELECT 时读到的旧空值），`is_ai_related` 与各评分基于正文而非仅标题

#### 场景:任一判分调用链的事件都先补全再判分
- **当** 实时告警链（或任何未来的第三个调用方）调用 `scoreUnscoredEvents` 判分一批空正文事件，该链**未**自行编排任何补全步骤
- **那么** 这些事件仍在送 LLM 之前各经一次补全尝试、成功者以正文 grounding 判分；**不存在**「被某条链先 title-only 判掉、从此永久离开补全工作集」的路径

#### 场景:正文补全失败时如实回退仅标题
- **当** 某事件正文补全失败（抓取失败 / og 缺失 / 命中全站样板 / SSRF 拒绝）、`content` 仍为空
- **那么** 判分**照常执行**、回退为仅标题输入（不伪造正文、不跳过该事件的判分），输出仍经 Zod 校验落库
