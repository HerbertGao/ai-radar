## Why

Model Radar 的三档抓取链只 detect 指纹变化并把源 flag 成 `pending`——**当 browser 档源 URL 本身失效（404 / 重定向到首页 / 登录墙变体 / 厂商重构路径）时，flag 不会告诉人「新 URL 在哪」**，人需要手动搜索厂商站点找新地址。Kimi 最近完成 `moonshot.cn → kimi.com` 域名迁移并预告会员体系重构，正是该问题的真实样本；目前 2 个 browser 档源（Z.ai、Kimi）规模可控但人工成本不可降，且随厂商路径重构频率上升会持续累加。本变更引入一个**严格范围（browser 档 + 同 allowlist 域内 drift）**的 evaluator-optimizer agent（「evaluator-optimizer」是 council A.position 既定术语指 agent 类型——评估 flag + 输出候选的语义判断 agent，**非** evaluator-optimizer 循环架构；见 design D8 澄清），把人工「找新 URL」环节的部分判断交给 LLM，候选仍经既有 Telegram HITL（Human-In-The-Loop，人在环）gate 批准——人仍是唯一事实写入者。

> Council C4 ruled fact：**agent autonomous discovery 与 allowlist 安全约束逻辑冲突**——agent 不能自由发现新域名并抓取（PR 评审维护的 allowlist 不可绕过）。本变更的调和方式是**把 agent 范围限死在「同一 vendor 已 allowlist 域内 URL drift」**：跨域/新域 drift 一律 escalate 到人工走 PR 流程，agent 不参与。

**术语说明**（首字母缩写与 council 编号首次展开）：
- **HITL**：Human-In-The-Loop，人在环——指 Telegram 一键批准 gate（人审 agent 候选后落库）
- **CAS**：Compare-And-Swap——指 `UPDATE … WHERE status='pending' AND …` 原子条件更新（claim token 用、防并发重复批）
- **CSPRNG**：Cryptographically Secure Pseudo-Random Number Generator——指 `node:crypto.randomBytes(16)` 生成 token（128-bit 熵、不可猜测）
- **TTL**：Time-To-Live——指 `extracted_at > now() - make_interval(hours => $ttl)` 过期窗口（默认 72h、过期不可批）
- **SSRF**：Server-Side Request Forgery——服务端请求伪造；本变更中指 agent 不可调 `safeFetch` / approve 侧 MUST 经 `assertUrlAllowed` 校验 / `set-source-url.ts` 内部 `assertUrlAllowed` 的安全约束（防 agent 被诱导抓取内网/私网/非 allowlist 域）
- **LLM**：Large Language Model——本变更中 agent 的语义判断引擎（Vercel AI SDK `generateObject` 调用）
- **Council C4**：本变更前置 council 评议会（council skill 召集）的第 4 项 ruled fact（不可推翻的事实裁决）——「agent autonomous discovery 与 allowlist 安全约束逻辑冲突」；C4 编号是 council 协议的议程项编号、非本仓库文件引用
- **council A.position**：同一 council 评议会中独立专家 seat A 的 position（立场声明）——「narrowly-scoped evaluator-optimizer agent」是 agent 类型名（评估 flag + 输出候选的语义判断）、非 evaluator-optimizer 循环架构

（通用库/协议名如 BullMQ / grammY / drizzle / Zod / SDK / JSON / SQL / HTTP / URL / DNS / IP / Cron / PR / DB / FK 见各自文档、不在此展开；首次出现处以「库名 + 一句用途」形式注明）

## What Changes

- **新增能力 `model-radar-url-drift-agent`**：当 browser 档源被 `setReviewFlag` 标为 `pending` 时，独立 BullMQ lane 触发 LLM evaluator-optimizer agent；agent 输入仅限 `mr_source` 行（含 `source_id` / `source_url` / `vendor_id` / `fetch_strategy`）+ flag `reason` + 该 vendor 在 `MR_SOURCE_DOMAIN_ALLOWLIST` 内已 allowlist 的域名清单（最小化信息），输出结构化候选 `{kind, candidate_url, confidence ∈ {low/medium/high}, reason}`（candidate 分支）或 `{kind, escalate_reason}`（escalate 分支）——`source_id` / `old_url` 是 agent **输入**侧字段（非输出字段，见 design D8 schema：`old_url` 经 schema 工厂闭包注入 refine 用、`source_id` 仅作 propose 侧关联、不进 refine），**MUST 经 Zod schema 校验**。
- **关键安全约束（C4 调和）**：候选 `candidate_url` 的 host **MUST** 已在该 vendor 的现有 allowlist 域名集合内（仅允许同 vendor 同 allowlist 域内 drift）；agent 不直接抓取候选 URL（避免绕 allowlist / SSRF 闸门），候选 URL 的验证留给既有 browser-scrape 链路下一轮（带 SSRF + allowlist chokepoint）。
- **复用既有 Telegram HITL gate**：新增 `mr_url_drift_review` 表（schema 与 `mr_price_review` 对称：CSPRNG token、单 plan/source partial unique、TTL、**frozen 字段集**（指 `old_url`/`candidate_url`/`confidence`/`reason`/`flag_opened_at` 在 open 时写入后 approve 侧只读不改——非新增名为 `frozen` 的列、而是字段集在 open 时冻结的语义约束））；新增 Telegram callback op 前缀 `mrud`（与既有 `mrpr` 并列）。Agent 自身不写任何 `mr_*` 事实。
- **Authorized setter（唯一事实写入口）**：人批准后，由 approve 路径调用新增 `setSourceUrl` 改 `mr_source.source_url`——同事务并把「以该 source 为 canonical 源」的 plan 的 `source_url` 对齐迁移到新 URL（维持既有 `plan.source_url ↔ mr_source.source_url` 对齐契约，见「非目标」D-B1 条），**仅改 URL、不碰价格/额度/兼容事实**，落在 approve.ts 范式（curation 区唯一允许 import fact writer 的文件）。Approve 侧 **MUST 防御性再校验** candidate host ∈ 该 vendor 的 allowlist 域集（vendor-scope，越界抛 `CrossDomainDriftError`；理论上 agent 不会输出越界候选，approve 侧兜底防 prompt injection 提权）。
- **ESLint 守卫**：`src/mr/scrape/url-drift-agent.ts` 与 agent lane 的 curation-side proposer **MUST NOT** import `src/mr/write/**`、`src/mr/ingest/**` 的 fact writer（含 `set-source-url.ts` authorized setter，仅 `approve.ts` 可 import）；agent 输出结构化 JSON 经 schema 校验后由 curation lane 写候选行（仿 `fingerprint.ts` 既有的 `no-restricted-imports` 范式——ban 的是 fact writer、**不** ban `src/db/index.ts`，见 design D9：`propose.ts:22` / `price-review-store.ts:21` / `fingerprint.ts:19` 均已 import 此 db handle、本变更镜像文件须保持对称）。
- **Prompt injection 缓解**：agent 输入的 `reason` 字段（来自 `setReviewFlag` / `compareAndUpdateFingerprint` 写入的开发者写死字符串）当作不可信文本——output schema 严格校验、候选 URL MUST 经 SSRF allowlist 闸门才能落到候选行、agent 不接受任何外部网页正文输入。
- **离线评估套件（precision 一词只属离线 eval）**：离线 eval ≥20 带 ground-truth 标签 case，覆盖真实 drift（Kimi `moonshot.cn → kimi.com` 历史样本）、no-op（URL 仍有效只价格变）、跨域 drift（agent 应 escalate 而非建议）、注入尝试（reason 字段塞 prompt injection）、SEO spam 页面；**precision 跌破 floor（`MR_URL_DRIFT_PRECISION_FLOOR`，默认 0.80）= CI job 失败（DD1）**——`test:eval` 断言失败 → CI job 变红（用真实 creds 本地/`workflow_dispatch` 跑；默认 CI 占位 creds 下干净 skip、绝不 green-claim），**非** ops-alert（`ops-alert-sink` 在 vitest 下被 `telegram.ts:43` 守卫降级、从 CI 发不出、离线 eval 不装配它）、不写 `mr_review_flag`——它是唯一能看见 agent 主导失败模式（同域、看着合理、实际错）的检测器；eval 与 agent 用**同一 dated snapshot 模型**（如 `openai/gpt-4o-mini-2024-07-18`，取代 cron）。
- **生产侧质量监控（不做 precision，(a) 两信号，见 design D7；均不写 `mr_review_flag`）**：① **采纳率/engagement 告警**——「连续 `MR_URL_DRIFT_ADOPTION_ROUNDS`（新 env、默认 3）轮 `total_candidates>0` 却 `adopted=0` → 经 `ops-alert-sink` 发 ops 告警（dedupKey `zero-adoption:url-drift`）」（分母只计 `total_candidates>0` 且 `adopted` 已回填的轮次；**`total_candidates` = `SELECT count(*) FROM mr_url_drift_review WHERE run_id = $run_id`**，从持久候选行重算、非 in-memory carded 计数器（DD2）；**delivery 须给 `runStaleness` 注入 `alert?: AlertSink`（DD3——当前它无 sink）**、受 `MR_STALENESS_ENABLED` 门控；显式标注 engagement/actionability，**非**正确性/precision）；② **离线 eval 红灯**——是 **CI job 结果**（precision 断言失败 → 红 job），**非** ops-alert（DD1、见上条）。（原 round-2 的 re-flag SLI 第二信号已删——council #2。）

### 非目标（必须显式登记）

- **不**让 agent 修改 `MR_SOURCE_DOMAIN_ALLOWLIST`（runtime 改 allowlist 一律禁止，PR-review 维护铁律不破）。
- **不**让 agent 写 `mr_source` / `mr_plans` / `mr_price_history` / `mr_plan_*` 任何事实（agent 只 propose；approve.ts 范式唯一写事实入口）。
- **不写 `mr_plans` 的价格/额度/兼容事实**；但 URL drift 批准会同事务把「以该 source 为 canonical 源」的 plan 的 `source_url` 对齐迁移到新 URL（维持 5b 录入规范的 `plan.source_url ↔ mr_source.source_url` 对齐契约——否则 `propose.ts` 会把已修正的 source 判为非 canonical → 永久 escalate 该 plan；这是**维持**既有 provenance 不变量、非新增范围）。
- **不**让 agent 处理 http 档源（http 档源 SSRF + allowlist 已完整；URL drift 由人接管即可，规模可控且 http 档变更通常是指纹变而非 URL drift）。
- **不**让 agent 自由抓取候选 URL 验证（避免 agent 绕过 SSRF 闸门；验证留给既有 browser-scrape 链路下一轮触发）。
- **不**改 http 档 / manual 档 / 推送 / 价格抽取 / 推荐解释任何既有路径。
- **不**新增 Telegram bot 实例（复用既有 `telegram-callback.ts` 的 grammY bot，仅扩 op 前缀与回调路由）。
- **不**改 `model-radar-recommender`、`model-radar-compare-api`、`model-radar-compare-web`、`model-radar-answer-first-web`、`model-radar-web-public-exposure`。
- **不**改 `MR_SOURCE_DOMAIN_ALLOWLIST` 的维护范式（仍 PR-review、check-in 常量；本变更不引入 DB-backed / env-var / hot-reload 任何动态 allowlist 路径）。
- **不建 `url-conflict` 的 suppress/terminal-escalate 回路（DD4，诚实登记的残余）**：approve catch ④ 对 `mr_source` UNIQUE(vendor_id, source_url) 冲突只分类 `reason:'url-conflict'` + fail-safe + 日志，agent 会每轮重提同候选；此冲突**当前即理论可达**（`UNIQUE(vendor_id, source_url)` 跨所有 fetch_strategy——Kimi 单 vendor 已带 http sibling `platform.kimi.com` + browser `www.kimi.com`，browser 源漂移到 sibling URL 即撞；RC 复核纠正原「不同 vendor 不可达」误判）；不建回路的真实依据是**概率极低 + fail-safe**（agent 只喂 vendor 域名、不喂 sibling URL，须凭空提出无关 sibling 路径 + 人须先误批 + 撞则 `apply_failed` URL 不改），非「结构不可达」，见 design 残余。

## Capabilities

### New Capabilities

- `model-radar-url-drift-agent`: Browser 档源 URL-drift 检测的 evaluator-optimizer agent——LLM 结构化**输出**候选 `{kind, candidate_url, confidence, reason}`（candidate 臂）或 `{kind, escalate_reason}`（escalate 臂）；`source_id` / `old_url` 是 agent **输入/闭包字段**（`old_url` 经 schema 工厂闭包注入 refine、`source_id` 仅 propose 侧关联，**均非 LLM 输出**，见 design D8）。候选 host MUST 在该 vendor 的现有 allowlist 域名集合内；候选经既有 Telegram HITL gate 批准后由 authorized setter 改 `mr_source.source_url`。Agent 不写事实、不改 allowlist、不抓取候选 URL。

### Modified Capabilities

- `model-radar-ingestion`: 既有「三档抓取仅做变更检测」需求补一句——**browser 档源 URL drift 由独立 agent lane 处理，agent 不写事实、不改 allowlist，候选经 HITL gate 批准后由 authorized setter `setSourceUrl` 改 `mr_source.source_url`**；http/manual 档源 URL drift 仍由人接管。新增「`mr_source.source_url` 的授权写入口 `setSourceUrl`」需求条目——`setSourceUrl` 是 authorized setter（仅 `approve.ts` 可 import，与既有 `setPlanAvailability` 的「authorized setter」性质对称），但**事务范式不同**：`setSourceUrl` 接受已开事务（`tx` 参数，approve 侧「claim + setSourceUrl」同事务原子），`setPlanAvailability` 自开事务（独立 plan 写、不与 claim 同事务）——见 spec ingestion 需求条目「事务范式」段。同事务内 `setSourceUrl` 还带 **old-URL CAS**（`WHERE source_url=oldUrl`，0 行 = 源已被更新的信号替换 → **抛 `StaleUrlError`** 走 apply-failed）、**对齐同 canonical 源 plan 的 `source_url`**、**`resolveFlag` 传 `expectedOpenedAt`（frozen `flag_opened_at`，防旧复核 resolve 掉卡片签发后新打的 flag generation；generation 不匹配时容忍不抛、URL 照常提交、新 flag 留 pending）**。
- `model-radar-price-curation`: 既有「mr_price_review 待批记录——令牌即一次性能力、带有效期」需求扩展为**通用 capability token 机制**——`mr_url_drift_review` 表与 `mr_price_review` 表对称（CSPRNG token、TTL、**frozen 字段集**（open 时冻结 `old_url`/`candidate_url`/`confidence`/`reason`/`flag_opened_at`、approve 侧只读不改——非名为 `frozen` 的列）、partial unique）；Telegram callback 路由按前缀 `mrpr` / `mrud` 分流，approval 侧 money-path red line（parse → authorize → channel-bind → apply）逐字沿用。

- **依赖**：零新第三方依赖（Vercel AI SDK + Zod + grammY + BullMQ 均既有）。
- **配置**：新增 env `MR_URL_DRIFT_ENABLED`（默认 `'false'`，仿 `MR_SCRAPE_ENABLED`）、`MR_URL_DRIFT_CRON`（默认 `33 9 * * 1`，错峰 browser scrape 09:17 之后）、`MR_URL_DRIFT_JOB_ATTEMPTS`（默认 3）、`MR_URL_DRIFT_CONFIDENCE_THRESHOLD`（默认 `medium`，低于此值不进 HITL gate 直接 escalate 不打扰人）、`MR_URL_DRIFT_TTL_HOURS`（默认 72，与 `mr_price_review` TTL 同范式）、`MR_URL_DRIFT_PRECISION_FLOOR`（默认 0.80，**离线 eval** precision 跌破 = `test:eval` 断言失败 → CI job 变红、**非** ops-alert、非阻塞，见 DD1/D7）、`MR_URL_DRIFT_ADOPTION_ROUNDS`（默认 3，engagement 信号「连续 N 轮零采纳」的 N，见 design D7）。**cron TZ 复用既有 `MR_SCRAPE_CRON_TZ`**（不新增独立 TZ env，使 D4「错峰于 browser scrape 之后」由构造保证、不受两个 TZ env 漂移影响；与 `curation-queue.ts` 同范式）。
- **数据模型**：新增表 `mr_url_drift_review`（schema 与 `mr_price_review` 对称：`id / source_id / run_id / old_url / candidate_url / confidence / reason / flag_opened_at / token / status / extracted_at / decided_at / decided_by / created_at`——`run_id` 是 URL drift 路径特有列、价格路径无对应字段，用于 `mr_url_drift_metric.run_id` 回填 join；`flag_opened_at` 亦是 URL drift 路径特有列、open 时从当前 source flag 的 `opened_at` 冻结读入、approve 侧作 `resolveFlag` 的 `expectedOpenedAt` 用（TOCTOU 防护）；partial unique `(source_id) WHERE status='pending'`、UNIQUE(token)；**回到 mr_* 零-CHECK 惯例、无 apply_failure_kind 列**）。新增表 `mr_url_drift_metric`（`id / run_id / total_candidates / adopted / ran_at`——`run_id` **UNIQUE**（每 run 一行、`ON CONFLICT(run_id) DO UPDATE` upsert 幂等、`run_id` = BullMQ job 稳定 id 跨 attempts 不变，Codex#5）；**`total_candidates` = `SELECT count(*) FROM mr_url_drift_review WHERE run_id = $run_id`**（从持久候选行重算、非 in-memory carded 计数器，DD2）；`adopted` nullable、人审批后回填 = 该 run 中 `status='approved'` 的候选数；详见 design D7 metric 字段表）。
- **行为变化**：仅 browser 档源被 flag 后的下一轮 drift lane 触发可见——agent propose 候选、Telegram 卡片推送、人批准后 `mr_source.source_url` 被改、下一轮 browser scrape 用新 URL。未触发路径（http/manual 档、未 flag 的 browser 源）逐字节不变。
