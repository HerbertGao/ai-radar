## MODIFIED Requirements

### 需求:飞书自定义机器人通道推送

系统必须提供一个飞书推送通道，基于飞书自定义机器人 webhook（原生 fetch + 签名）把每日日报/产品发现/告警推送到配置的飞书群。**飞书是可选通道，不破坏 P1 纯 Telegram 部署的向后兼容**：未配置飞书（无 `FEISHU_WEBHOOK_URL` / `FEISHU_SIGN_SECRET`）时，飞书**不纳入「已配置通道」集**、不参与分发、不报错——「向所有已配置通道分发」（daily-intel-pipeline）据此只发已配置的通道（至少 Telegram）。仅当**配置不完整**（声明启用飞书但 webhook URL 或签名密钥缺其一）时，env 校验必须以明确错误快速失败，禁止用空值发送或静默半启用。消息必须渲染为飞书 JSON 卡片（区别于 Telegram 的 MarkdownV2），卡片内跳转只用按钮/文字链方式（飞书自定义机器人不支持点击回调到服务端，禁止设计依赖回调交互的卡片）。所有外部调用必须带重试与错误日志，发送失败禁止静默吞掉。

**卡片内 lark_md 文字链的 URL 段必须用独立 URL 转义器对 `(`/`)`/`[`/`]`/`\` 做 percent-encode（`%28`/`%29`/`%5B`/`%5D`/`%5C`），禁止裸插、禁止复用文本转义器。** 卡片跳转用 lark_md 内联链接 `[文本](url)`，URL 取确定性 `canonical_url` / `canonical_source_url`（来自外部 feed、可含字面 `(` `)` `[` `]` `\`，如 `Go_(programming_language)`、带括号 query 的 URL）。lark_md 按 markdown 语法解析 `(...)` 段：URL 内未转义的 `)` 会在第一个 `)` 处**提前闭合链接**、把剩余 URL 当普通文本泄漏、链接指向被截断的错误地址、破坏卡片渲染与可点击性。故飞书全部 lark_md 渲染器（事件 / 产品 / 周报 / 经验，及任何后续新增的 lark_md 链接行）的 URL 段**必须经独立 URL 转义函数处理**——对 `(`/`)`/`[`/`]`/`\` 做 percent-encode，使链接结构对任意 lark_md 解析模型（无论「首个 `)` 即闭合」还是「CommonMark 配对括号」）都不被破坏（correct-by-construction、与解析器无关），与 telegram-push「链接 URL 含特殊字符用独立规则转义」的 URL 专用转义对称。**为何同时编码 `(`/`[`**：只编码闭括号 `)`/`]` 会在配对模型下留下不配对的开括号反而破链，故开闭都编。**禁止复用飞书文本转义器**（转 `[ ] ( ) \`，为链接文本/标题/正文设计）：把它作用在 URL 上会过度转义 URL 内合法字符、改变跳转目标。`%28`/`%29`/`%5B`/`%5D`/`%5C` 是合法 URL 编码，点击后解码回原字符、跳转到原 URL（含其中的 `)`）、href 不变。

飞书自定义机器人限流为单租户单机器人 100 次/分钟、5 次/秒；系统调度推送时必须避开整点/半点（如 10:00、17:30）等高压时刻以降低 11232 限流错误概率。P2 各推送路径（日报每日一条、产品/周报低频、告警高频但单条）天然稀疏，单条卡片发送远低于 5 次/秒，**P2 不实现发送侧节流闸**；但若未来同一秒并发多条卡片（如批量分条），必须补 5 次/秒发送节流——本期记为已知边界、不实现。

**「避整点」的判据 MUST 为【分钟字段展开后的分钟集合 ∩ {0, 30} = ∅】，MUST NOT 为字面量比较（本需求修改点）**：判据的**主语是展开集，不是字段字符串**。「分钟字段的默认值**字符串** ∉ {'0','30'}」这一形式的判据，对每一种含步进或列表的 cron **恒真**——`*/15` 展开为 `{0,15,30,45}`（**同时**撞整点与半点）却因字符串 `'*/15' ∉ {'0','30'}` 判过；`*`（每分钟都发，必撞整点半点）同样判过。这不是理论缺口：`ALERT_SCAN_CRON` 的默认值 `*/15` 违反本规则**长期未被发现**，正因为守卫按字面量比较、**从未展开过任何表达式**——一个恒绿的守卫等于没有守卫。

**展开语法 MUST 覆盖 cron 分钟字段的全部合法形式（缺一即重开上述缺口）**：

| 形式 | 展开 |
|---|---|
| `*` | `0`–`59` 全集 |
| `*/n` | `{m ∈ [0,59] | m mod n == 0}` |
| `a-b` | `a` 到 `b` 的连续整数 |
| `a-b/n` | 自 `a` 起步进 `n`、不超过 `b` |
| **`a/n`** | **隐式 `a-59/n`**（自 `a` 起步进 `n` 到 59） |
| `a,b,c` | 各项展开的并集 |
| 纯数字 | 单元素集 |

判定即「该并集与 `{0, 30}` 是否有交」。据此：`*/15` → `{0,15,30,45}` → **违反**；`*/20` → `{0,20,40}` → **违反**（含 0）；**`0/15` → `{0,15,30,45}` → 违反**；`4-59/15` → `{4,19,34,49}` → **合规**；`9-59/20` → `{9,29,49}` → **合规**；`3` → `{3}` → 合规。

**无法展开的分钟字段 MUST 判为违反（fail-closed，绝不可省）**：展开器遇到不属于上表任一形式的分钟字段时，**MUST 抛错**，且判据侧 **MUST 把「抛错」计为违反**，**MUST NOT** 静默返回空集。返回空集会使 `∅ ∩ {0,30} = ∅` **恒成立 → 判过**——一个「解析不出就当合规」的守卫，正是本判据整节论证要杀死的那类空守卫。**`a/n` 的遗漏即是此类**：`cron-parser`（BullMQ 所用）支持它，而只列 `*/n` / `a-b/n` 的文法会对 `0/15` 各项皆不匹配 → 空集 → 判过，然而它实际展开为 `{0,15,30,45}`、撞整点半点。展开器对**数值非法**的字段同样 MUST 抛错（分钟值越界如 `60`、步进 ≤ 0 如 `*/0`、倒置区间如 `30-10`）——数值有效性是文法的一部分，非法即违反，绝不静默空集。

**覆盖面 MUST 为【全部会触发飞书发送的定时 cron】的默认值，MUST NOT 只断言其中一个（绝不可省）**：单条断言的失败模式已被实证——它只查 `DAILY_DIGEST_CRON`，而当时唯一违反规则的恰是它**没查**的那条。故断言 MUST 遍历下列**全部**默认值，任一违反即失败：

- `DAILY_DIGEST_CRON`（日报，含新闻要闻段与产品新品段）；
- `ALERT_SCAN_CRON`（实时告警扫描；其默认值与生产覆盖值的约束见 realtime-alerts「重大发布事件级实时告警」）；
- 周报 cron 默认值 `DEFAULT_WEEKLY_CRON`（`src/pipeline/weekly-queue.ts` 的**模块常量、非 env 变量**——断言 MUST 直接读该常量，**MUST NOT** 因「它不在 env 里」而漏掉它，**MUST NOT** 在测试里抄一份字面量副本）；
- `MR_PRICE_CURATION_CRON`（Model Radar 价格复核提案卡片，经飞书通道发出）。

**不在覆盖面内的定时 cron（不经飞书发送）**：`MR_EVENT_REVIEW_CRON`、**`MR_SCRAPE_HTTP_CRON` 与 `MR_SCRAPE_BROWSER_CRON`（Model Radar 抓取是【两条】独立 cron，不是一条）**、`MR_STALENESS_CRON`。

新增任何会触发飞书发送的定时 cron 时，MUST 同时把它加进该断言的覆盖清单——覆盖清单是本判据唯一的强制手段。

**机械断言只覆盖默认值；生产覆盖值的合规由部署步骤强制（诚实边界，MUST 写明）**：断言读的是代码里的默认值，**读不到生产 env 的覆盖值**。故凡改写了某条 cron 默认值的变更，MUST 在同一次部署里同步改写该 cron 的**生产覆盖值**——否则断言全绿而生产照撞整点，判据在生产上失效。本规范在此显式登记该边界，系统不为此增设「从生产 env 反查合规」的运行时闸（那会把一个部署纪律问题变成一个启动期硬失败）。

**判据约束的是【触发时刻】，不是发送时刻（诚实边界，MUST 写明，防因果被写死）**：cron 只是链路的**起点**，实际发卡在采集 → 塌缩 → 判分 → 回填 → 选候选 → 摘要之后**数分钟**。故本判据是**降低撞上飞书全网高峰概率**的启发式约束，MUST NOT 被表述为「发送时刻必不落整点/半点」的保证——系统不提供该保证，也不为此增设发送时刻的对齐逻辑（那会为一个概率性限流问题引入确定性复杂度）。

#### 场景:推送渲染为飞书 JSON 卡片
- **当** 飞书通道推送一份日报
- **那么** 消息以飞书 JSON 卡片格式发送、跳转用按钮/文字链，发送成功

#### 场景:飞书链接 URL 含特殊字符用独立规则转义
- **当** 某飞书卡片（事件/产品/周报/经验任一渲染器）的来源 URL 含字面 `(`、字面 `)`、字面 `[`、字面 `]`、字面 `\`、以及常见的 `.`/`-`/`_`/`=`/`?`/`/`
- **那么** URL 经独立飞书 URL 转义函数处理：`(`/`)`/`[`/`]`/`\` 被 percent-encode 为 `%28`/`%29`/`%5B`/`%5D`/`%5C`、而 `.`/`-`/`_`/`=`/`?`/`/` **原样不变**（不复用文本转义器、不加反斜杠）；URL 段内无裸 `(`/`)`/`[`/`]`/`\`，链接结构对任意 lark_md 解析模型都完整（不在首个 `)` 处提前闭合、也不留不配对 `(`），可点击且解码后跳转到完整含 `)` 的原 URL（href 不变）

#### 场景:未配置飞书时纯 Telegram 部署照常启动
- **当** P1 部署升级到 P2 但未配置任何飞书 env（webhook/secret 均空）
- **那么** 飞书不纳入「已配置通道」集、不报错，系统照常启动并只向 Telegram 分发，向后兼容

#### 场景:飞书配置不完整时启动即报错
- **当** 声明启用飞书但 webhook URL 或签名密钥仅配其一（不完整）
- **那么** 系统以明确错误信息快速失败，禁止用空值发送或静默半启用

#### 场景:定时触发避开整点降低限流
- **当** 逐一检视全部会触发飞书发送的定时 cron 默认值（`DAILY_DIGEST_CRON`、`ALERT_SCAN_CRON`、周报模块常量 `DEFAULT_WEEKLY_CRON`、`MR_PRICE_CURATION_CRON`），把各自的分钟字段按 `*` / `*/n` / `a-b` / `a-b/n` / `a/n` / `a,b,c` / 纯数字**展开为分钟集合**
- **那么** 每一个展开集与 `{0, 30}` 的交集为空（如 `3 8 * * *` → `{3}`；`4-59/15 * * * *` → `{4,19,34,49}`；`7 9 * * 1` → `{7}`；`53 9 * * *` → `{53}`），机械可断言不落整点/半点，以降低 11232 限流错误概率

#### 场景:步进式 cron 的展开集撞整点被判违反
- **当** 某会触发飞书发送的 cron 默认值取 `*/15 * * * *`（或 `*/20`、或 `0/15`、或 `*`）——其分钟字段**字符串**既不等于 `'0'` 也不等于 `'30'`
- **那么** 判据仍判其**违反**：展开集 `{0,15,30,45}`（`*/20` → `{0,20,40}`；`0/15` → `{0,15,30,45}`；`*` → 0–59 全集）与 `{0, 30}` 有交——**绝不**因字面量不等于 `'0'`/`'30'` 而放行（字面量判据正是在此处恒绿、漏掉了 `ALERT_SCAN_CRON` 的违反）

#### 场景:无法展开的 cron 分钟字段判为违反而非放行
- **当** 判据遇到一个不属于既定文法（`*` / `*/n` / `a-b` / `a-b/n` / `a/n` / `a,b,c` / 纯数字）的分钟字段
- **那么** 展开器**抛错**、判据把该 cron 判为**违反**（fail-closed）——**绝不**静默返回空集：空集会使「与 `{0,30}` 无交」恒成立而判过，等于把一个未知语法**默认当成合规**。`a/n` 的漏列正是此类失效（`0/15` 实际展开为 `{0,15,30,45}`，却因文法不覆盖而被空集判过）
