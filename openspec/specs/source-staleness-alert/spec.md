## 需求
### 需求:按源陈旧度检测（确定性 DB 判定）
系统必须对每个待监控源，基于 `raw_items` 按 `source` 聚合的最近入库时间（`max(fetched_at)`）判定该源是否「连续超过其阈值天数零新增」。判定必须完全由确定性 DB 查询得出，**禁止**交由 LLM 判断。一个待监控源，若其最近 `raw_items` 入库时间早于「参考时刻 − 阈值天数」，或从未有任何 `raw_items`（`max(fetched_at)` 为 NULL），必须被判定为**陈旧**。

#### 场景:某源超阈值零新增判为陈旧
- **当** 某源最近一条 `raw_items` 的 `fetched_at` 早于「参考时刻 − 该源阈值天数」
- **那么** 该源必须被判定为陈旧

#### 场景:某源阈值内有新增判为新鲜
- **当** 某源在「参考时刻 − 该源阈值天数」之后仍有 `raw_items` 入库
- **那么** 该源不得被判定为陈旧

#### 场景:从未产出任何条目的源判为陈旧
- **当** 某待监控源在 `raw_items` 中没有任何行（`max(fetched_at)` 为 NULL）
- **那么** 该源必须被判定为陈旧（视为长期零新增，对应"某源从未出数"的失效形态）

#### 场景:采集抛错被隔离吞掉的源仍能被判陈旧
- **当** 某源的采集连续多轮抛错、被 collector 层 `Promise.allSettled` 隔离吞掉（其 `max(fetched_at)` 停在上次成功之日、超过阈值）
- **那么** 该源必须被判定为陈旧（"抛错静默死"形态由 max 停滞捕获，不依赖任何采集层错误信号）

#### 场景:采集成功但返回 0 条的源仍能被判陈旧
- **当** 某源的采集连续多轮成功返回但 0 条（不抛错、其 `max(fetched_at)` 同样停在上次有产出之日、超过阈值）
- **那么** 该源必须被判定为陈旧（"返回 0 静默死"形态与"抛错"在 DB 层同归为 max 停滞，统一捕获）

### 需求:陈旧源经既有 AlertSink 上报
系统检测到任一陈旧源时，必须经**既有 `AlertSink`** 上报告警，告警内容必须可定位到具体源（至少含源名与该源的最近入库时间或零新增天数）。无任何陈旧源时**禁止**发出该告警。

#### 场景:存在陈旧源则告警
- **当** 一个或多个待监控源被判定为陈旧
- **那么** 系统必须经 `AlertSink` 上报，且告警内容包含每个陈旧源的源名与其陈旧信息

#### 场景:全部新鲜则静默
- **当** 所有待监控源均未被判定为陈旧
- **那么** 系统不得发出陈旧度告警

### 需求:阈值 env 可配且可按源覆盖
陈旧度阈值（天数）必须由 env 提供**全局默认值**（`SOURCE_STALENESS_ALERT_DAYS`），并且必须支持**按源覆盖**，使不同采集节奏的源可设不同阈值。某源未配置覆盖时，必须使用全局默认阈值。阈值**禁止**硬编码在判定逻辑内。

#### 场景:未覆盖的源用全局默认阈值
- **当** 某源没有配置按源覆盖阈值
- **那么** 该源的陈旧判定必须使用全局默认阈值 `SOURCE_STALENESS_ALERT_DAYS`

#### 场景:已覆盖的源用其自身阈值
- **当** 某源配置了按源覆盖阈值
- **那么** 该源的陈旧判定必须使用其覆盖阈值，而非全局默认

### 需求:覆盖配置解析为容错跳过而非 fail-fast
按源覆盖串（`source:days` 列表）的解析必须**容错**：任一**误配项**（天数非正整数、天数非整数、缺 `source` 或缺 `days`、`source` 不在已注册源集合内）必须被**跳过并记日志**（日志提示误配以便运营发现），不得使整个应用启动失败、不得使该项静默生效为错误阈值。**纯空串项**（如多余逗号 `a,,b`、首尾逗号产生的空段）是良性格式、**静默跳过不记日志**（避免噪音）。同一 `source` 出现多次时必须有确定性解析（后者覆盖前者，last-wins）。此处**有意偏离** env 层其它配置的 fail-fast 风格——陈旧度告警是 advisory 可观测能力，一条坏覆盖不应拖垮整个应用启动。

#### 场景:非正整数天数被跳过
- **当** 覆盖串含 `blogger:0` / `blogger:-1` / `blogger:abc`（天数非正整数）
- **那么** 该项必须被跳过并记日志，`blogger` 回退到全局默认阈值（禁止让 0/负数阈值生效致该源恒判陈旧）

#### 场景:未知源名被跳过并告警
- **当** 覆盖串含 `blooger:7`（拼写错误、不在已注册源集合内）
- **那么** 该项必须被跳过并记日志（提示该键不匹配任何已注册源），不得静默使真实 `blogger` 沿用默认而运营以为已配置

#### 场景:重复源键 last-wins
- **当** 覆盖串含 `blogger:7,blogger:2`（同源多次）
- **那么** 解析结果必须确定性地取 `blogger:2`（后者覆盖前者）

### 需求:检测为 best-effort，绝不影响主流程
陈旧度检测必须作为 best-effort 阶段运行：其任何异常必须被隔离（仅记日志），**禁止**向上抛断、**禁止**影响日报工作流的 `outcome`、**禁止**计入任何降级/熔断分母、**禁止**影响推送。

#### 场景:检测异常被隔离
- **当** 陈旧度检测的查询或判定过程抛出异常
- **那么** 异常必须被捕获并记日志，日报工作流的 `outcome` 与推送不受影响

#### 场景:检测不进入熔断分母
- **当** 陈旧度检测在日报工作流内运行
- **那么** 其结果（无论是否发现陈旧源）不得计入 value-judge / digest 等阶段的降级或熔断分母

### 需求:监控源全集取自 collector registry，但排除结构性停用源
待监控源的全集必须取自 collector registry 的已注册源（`source` 列表），使新增 collector 后其源**自动纳入**陈旧度监控，无需改动告警代码。但必须**排除本部署结构性不产出的源**——即因 list 型配置为空而其 collector 恒返回 0 条的源：`RSS_FEEDS` 空的 `rss`、`BLOGGER_FEEDS` 空的 `blogger`、`SITEMAP_SOURCES` 空的 `sitemap`（三者各有独立空配置停用开关；其余已注册源无此类「空配置即恒不产出」开关，恒纳入）。因为这类源 `max(fetched_at)` 恒为 NULL、会被"从未产出→陈旧"判定为**每日永久误报**，而按源阈值覆盖对 NULL 无效（NULL 绕过天数比较），故必须在源集合层排除。「配置非空却 0 行」（真失效）与「配置为空」（有意停用）由配置是否非空区分。

#### 场景:新增源自动纳入监控
- **当** 向 collector registry 注册了一个新的采集源（且其未被结构性停用）
- **那么** 该新源必须在下一次陈旧度检测中被自动纳入（若其满足陈旧条件即触发告警），无需修改陈旧度检测代码

#### 场景:配置为空的源被排除出监控
- **当** 某源因配置为空而恒不产出（如 `RSS_FEEDS` 为空使 `rss` collector 恒返回 0 条）
- **那么** 该源必须被排除出待监控集合，不得因其 `max(fetched_at)` 恒 NULL 而每日永久告警

#### 场景:配置非空却零产出的源仍被监控并告警
- **当** 某源配置非空（如 `BLOGGER_FEEDS` 已配 feeds）却在 `raw_items` 中零行（真失效，对应本轮 blogger 从未出数的 bug）
- **那么** 该源必须留在待监控集合且被判陈旧告警（结构性停用的排除禁止误伤"配了却坏了"的真失效）
